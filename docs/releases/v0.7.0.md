# v0.7.0 — Reusable Silicon Evidence Gate

`v0.7.0` turns the repository from a standalone Ibex verification prototype into a reusable, commit-pinnable GitHub Action for evidence-gated RTL and firmware changes.

## Highlights

### Reusable GitHub Action

External repositories can now call the gate directly:

```yaml
- id: silicon_gate
  uses: safal207/ibex-agent-verification@v0.7.0
  with:
    request: artifacts/gate/gate-request.json
    report: artifacts/gate/gate-decision.json
```

The action publishes:

- `decision`: `ALLOW`, `BLOCK`, or `ESCALATE`;
- `reason_codes`;
- request and report SHA-256 values;
- the report path;
- a GitHub job summary with regression metrics and deterministic reasons.

Teams can start in observe-only mode with `fail_on_block: "false"` and `fail_on_escalate: "false"`, then enable enforcement through branch protection.

### Deterministic silicon evidence gate

The gate evaluates commit-bound evidence from:

- architectural trace comparison;
- baseline and candidate timing reports;
- baseline and candidate control-flow reports;
- an evidence manifest tied to the candidate commit.

It fails closed on architectural mismatches, evidence-binding failures, new unexplained timing anomalies, and policy-budget overruns. Valid evidence always produces one machine-readable decision.

### Hardware-backed demonstrations

The release includes three reviewable evidence paths:

1. generated `ALLOW` and `BLOCK` fixtures for the gate contract;
2. real firmware baseline/candidate runs on pinned Ibex Verilator;
3. clean and RTL-patched Ibex simulators running identical firmware.

The real RTL demonstration preserves architectural equivalence and candidate replay while blocking an intentional instruction-memory latency regression.

### Evidence integrity

- referenced evidence files are SHA-256 hashed;
- evidence paths cannot escape the request directory;
- AI-attributed changes can require an explicit model identifier;
- decision severity is deterministic: `BLOCK > ESCALATE > ALLOW`;
- raw traces, FST waveforms, normalized reports, manifests, tool versions, and decisions remain reviewable artifacts.

## Trust boundary

This release does not replace formal verification, CDC/RDC analysis, STA, power or physical sign-off, security review, coverage closure, or human tape-out authorization. It provides a deterministic evidence boundary for deciding whether an AI-generated or human-generated change may proceed to the next verification stage.
