# Official Ibex Instruction Trace Adapter

The adapter parses the human-readable log generated by `ibex_tracer`, normally named `trace_core_<HARTID>.log`.

## Pinned format reference

The initial parser is tested against the documented format at lowRISC/ibex commit:

```text
022f084096baed0a9b5ebdf697ed2965f13e8ed8
```

Relevant upstream files:

- `doc/03_reference/tracer.rst`
- `rtl/ibex_tracer.sv`
- `dv/uvm/core_ibex/riscv_dv_extension/ibex_log_to_trace_csv.py`

The upstream tracer documents tab-separated fields for simulation time, cycle, PC, instruction, decoded instruction, register accesses, and memory contents. The SystemVerilog implementation prints register reads as `xN:0x...`, register writes as `xN=0x...`, physical addresses as `PA:0x...`, and load/store values as `load:0x...` and `store:0x...`.

## Command

```bash
ibex-av parse-ibex-trace \
  --input trace_core_00000000.log \
  --output artifacts/architectural.jsonl \
  --metadata-output artifacts/metadata.jsonl \
  --timing-output artifacts/timing.jsonl \
  --report artifacts/parser-report.json
```

Outputs:

- `architectural.jsonl`: comparator-compatible PC, instruction, register write, and memory evidence;
- `metadata.jsonl`: simulation time, cycle, instruction width, decoded instruction, register reads, and source line;
- `timing.jsonl`: cycle gaps between consecutive retired instructions;
- parser report: source SHA-256, source line count, instruction count, and cycle range.

## Strictness

The parser skips only blank lines and the documented header. Any other unsupported line fails with source and line context. Cycles must increase strictly. The parser does not silently discard a line that might contain architectural evidence.

The parser preserves 16-bit versus 32-bit instruction width in metadata. The architectural comparator still receives the numeric instruction value because that is its current public contract.

## Timing interpretation

The instruction log is sufficient to measure a gap between retired instructions. It is not generally sufficient to prove why that gap occurred.

For example, a memory instruction followed by a three-cycle gap is not enough evidence to declare `MEMORY_WAIT`. The adapter marks only that a memory access occurred. Without `data_ready`, explicit wait counters, waveform transitions, or comparable causal signals, the timing analyzer must return `UNKNOWN`.

Instruction mnemonics such as `mul`, `div`, and `rem` may be forwarded as low-confidence instruction-class evidence for long-latency execution. This is still a hypothesis, not a confirmed root cause.

## Remaining integration work

A full end-to-end run still requires:

1. building the pinned or explicitly selected Ibex revision;
2. compiling and running a bare-metal test in Verilator;
3. retaining the original raw trace and tool-version manifest;
4. parsing the generated trace with this adapter;
5. adding a reference ISA oracle;
6. extracting causal signals from waveform or simulator instrumentation for stronger timing diagnosis.
